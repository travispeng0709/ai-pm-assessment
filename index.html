    AI 用量统一换算公式

    将不同 AI 渠道的资源消耗（Comate Token、付费渠道金额）统一换算为"AI 用量分"（0-100 分），量化各渠道的实际使用程度，便于横向对比。

    AI 用量分 = Comate 用量分 × W₁ + 付费渠道用量分 × W₂
    01

    统一计量单位

    AI 用量分（0-100 分），是所有 AI 资源消耗的统一计量单位。不同渠道的原始消耗量各自换算为 0-100 的分值，再按权重合并为最终用量分。

    核心思路：每个渠道的消耗量除以对应配额或基线，得到 0-100% 的使用率，即为该渠道的用量分。
    AI 渠道原始单位换算方式统一单位
    Comate Token 消耗Token消耗量 ÷ 个人配额AI 用量分
    付费渠道消耗（Cursor / API Key / Gemini / Coding Plan）元消耗金额 ÷ 额度池总额AI 用量分
    02

    总换算公式

    AI 用量分 = Comate 用量分 × W₁ + 付费渠道用量分 × W₂ 默认权重：W₁ = 50%，W₂ = 50%（可调，W₁ + W₂ = 100%） A 类员工（仅 Comate）：AI 用量分 = Comate 用量分 B 类员工（全渠道）：AI 用量分 = Comate 用量分 × W₁ + 付费渠道用量分 × W₂

    权重 W₁ 和 W₂ 互斥，总和始终为 100%。调整一个权重时，另一个自动补齐。

    考核方案 · 决策摘要

    产品经理 AI 应用能力考核方案

    不考核"谁消耗 AI 资源多"，而考核"谁能把 AI 转化为真实产品产出和可复用团队资产"。从"AI 使用量"转向"AI 产出质量"和"团队资产沉淀"。

    最终指数 = AI 使用行为分 × 40% + AI 有效产出分 × 60%
    01

    一句话结论

    本方案不考核"谁消耗 AI 资源多"，而考核"谁能把 AI 转化为真实产品产出和可复用团队资产"。

    建议先作为试运行指标运行 1-2 个周期，观察数据分布、执行成本和公平性，再决定是否正式纳入绩效。

    02

    为什么要做

    公司已为产品经理配置 Comate、Cursor、API Key、Gemini、Coding Plan 等 AI 资源。如果只看是否开通或消耗金额，无法判断 AI 是否真正提升了产品工作。

    本方案要解决三个问题：

    1. AI 资源是否真的被用起来。
    2. AI 是否提升了需求分析、方案设计、竞品研究、用户反馈分析等产品工作质量。
    3. 是否沉淀出可复用资产，例如 Agent、Skill、Prompt 模板、知识库、评测集、SOP。
    从"AI 使用量"转向"AI 产出质量"和"团队资产沉淀"。
    03

    总体考核框架

    最终指标：产品经理 AI 应用能力指数，满分 100 分。

    最终指数 = AI 使用行为分 × 40% + AI 有效产出分 × 60%
    模块权重看什么
    AI 使用行为分40%是否持续、真实地使用公司分配的 AI 资源
    AI 有效产出分60%是否形成真实产品产出、可复用资产和团队影响

    产出分占 60%，是为了避免把考核导向"刷 Token、刷额度、刷使用天数"。产品经理的核心价值不是用了多少 AI，而是用 AI 做成了什么。

    04

    有效产出看四类

    维度默认分值判断重点
    产品工作提效20是否减少重复劳动，提升 PRD、会议纪要、需求整理等交付效率
    产品分析质量25是否提升需求判断、竞品分析、用户反馈分析、数据解读质量
    可复用资产沉淀35是否沉淀 Agent、Skill、Prompt、知识库、SOP、评测集等资产
    团队复用与影响20是否被他人使用，是否进入团队流程或模板库

    高分不是看材料多少，而是看：

    1. 是否有真实业务场景。
    2. 是否有可核验证据。
    3. 是否被采纳、复用或进入团队流程。
    4. 是否能提升团队长期效率。
    05

    如何保证公平

    5.1 按资源权限分类

    不同员工 AI 资源权限不同，不能用同一套行为指标硬套。

    员工类型行为分计算原则
    仅 Comate 员工只看 Comate 活跃指数
    全渠道员工在付费渠道和 Comate 中取主渠道表现

    这样可以避免两类不公平：

    1. 仅 Comate 员工因为没有付费渠道而被扣分。
    2. 全渠道员工被迫每个渠道都象征性使用。

    5.2 按项目阶段允许替代证据

    不同产品经理所处项目阶段不同，不能只看是否进入正式评审或是否被多人复用。

    项目阶段可接受证据方向
    探索期机会点、用户访谈、竞品分析、方案假设被采纳
    交付期进入需求评审、版本规划、需求池或方案决策
    运营期分析结论被用于优化决策、问题归因或运营动作
    收尾期复盘、交接文档、SOP 被采纳并入库
    保密或独立项目脱敏摘要 + 负责人确认 + PMO 可核验原始材料
    证据形态可以因项目阶段不同而变化，但质量标准不降低。
    06

    如何防止刷量和套利

    风险保护机制
    刷 Token、刷额度行为分只占 40%，产出分占 60%
    几天内集中烧完额度付费渠道集中消耗封顶
    行为分高但无产出一致性门控，必要时行为分折算
    产出分高但无使用记录要求补充使用过程证据
    只有口头描述、无材料证据等级限制得分上限

    关键规则：

    1. 产出分低于 40 分，最终指数最高不超过 50 分。
    2. 行为分和产出分偏差过大，会进入一致性校验。
    3. 没有证据的产出不能评高分。
    4. 只创建资产但无人使用、无业务验证，不能拿资产高分。
    07

    落地方式

    建议分两层执行：

    层级看什么用途
    管理层方向、结果、风险、资源投入判断是否启动试运行，是否正式纳入绩效
    PMO / 产品负责人公式、阈值、证据、异常、申诉负责评分、校准、复核和落地执行

    月度执行建议轻量化：

    1. 普通案例按量表快速评分。
    2. 高分案例、异常案例、申诉案例重点复核。
    3. 团队级资产季度集中评审。
    08

    试运行建议

    建议先运行 1-2 个周期，不直接硬挂绩效。

    试运行期间重点观察：

    1. 数据是否能采集到。
    2. 员工提交证据是否容易核验。
    3. 各团队评分口径是否一致。
    4. 是否出现刷 Token、刷材料、刷资产数量。
    5. 评审成本是否可接受。
    6. 是否真的推动产品经理沉淀 Agent、Skill、Prompt、SOP 等资产。

    试运行结束后，再决定：

    1. 是否正式纳入绩效。
    2. 行为分和产出分权重是否调整。
    3. 证据等级和异常阈值是否需要优化。
    4. 是否建立团队级 AI 资产库。
    09

    关键决策事项

    决策项建议
    是否启动建议启动试运行
    是否直接纳入绩效不建议第一期直接硬挂绩效
    试运行周期建议 1-2 个考核周期
    考核导向产出和资产沉淀优先，资源消耗只作为辅助
    组织投入需要 PMO 支持数据报表、评分校准和复核
    后续机制建议同步建设团队级 AI 资产库

    员工分类

    员工类型权限情况行为分计算原则
    A 类：仅 Comate 员工只能使用 Comate不考核付费额度池，只看 Comate 活跃指数

    14

    AI 使用行为分

    14.1 A 类员工

    AI 使用行为分 = Comate 活跃指数

    14.2 B 类员工

    AI 使用行为分 = 主渠道使用分 主渠道使用分 = MAX(付费渠道使用分, Comate 活跃指数)

    产品经理可以根据任务选择最适合的 AI 渠道，不要求每个渠道都使用。

    14.3 B 类员工付费资源闲置保护

    若 B 类员工单月付费额度池使用率 < 20%，则仅触发资源适配预警，不直接扣分。 若连续 2 个考核周期付费额度池使用率 < 20%，且无合理低适配说明， 则当期 AI 使用行为分按 90% 折算。 若连续 3 个考核周期付费额度池使用率 < 20%， 则进入资源配置复盘，判断是否需要调整额度或员工类型。

    可接受的低适配说明：

    1. 本月主要工作为需求收敛、流程梳理、轻量文档协作，Comate 已能满足。
    2. 本月处于探索期或收尾期，付费模型使用价值较低。
    3. 涉及敏感数据、保密项目或渠道权限限制，暂不适合使用外部付费渠道。
    4. 团队已有共享资产或公共工作流，个人付费额度暂未形成必要投入。
    15

    付费渠道使用分

    适用于 B 类全渠道员工。

    付费渠道使用分 = 付费额度池使用率 × 50% + 付费有效活跃率 × 50%

    15.1 付费额度池使用率

    付费额度池使用率 = MIN(付费渠道总消耗金额 / 个人付费额度池总额, 100%) 付费渠道总消耗金额 = Cursor 消耗 + API Key 消耗 + Gemini 消耗 + Coding Plan 消耗

    执行要求：

    1. Cursor、API Key、Gemini、Coding Plan 共用一个个人付费额度池。
    2. 不按单个渠道分别计算独立使用率。
    3. 分母使用考核周期开始时公司分配的额度池。
    4. 使用率最高按 100% 计分，超出部分只用于资源扩容或异常分析。

    15.2 付费有效活跃率

    付费有效活跃率 = MIN(付费有效使用天数 / 月度目标活跃天数, 100%)
    月度目标活跃天数 = 8 天

    付费有效使用日定义：

    当天 Cursor / API Key / Gemini / Coding Plan 等付费渠道有实际消耗，并且对应真实产品工作场景，可计为 1 个付费有效使用日。

    过渡口径：

    当天付费渠道消耗金额 > 0，则计为 1 个付费有效使用日

    过渡口径约束：

    1. 每月付费有效使用天数上限为 15 天。
    2. 报表中必须标注该字段为过渡口径。
    3. 正式纳入绩效前，应至少具备"使用日 + 任务/案例/输出链接"中的一类可追溯证据。

    15.3 付费渠道集中消耗封顶

    为避免少数几天集中消耗拉高行为分，付费渠道使用分需受最低活跃门槛封顶。

    付费有效使用天数默认目标 8 天下的区间付费渠道使用分上限
    < 目标天数 × 30%0 - 2 天最高 60 分
    ≥ 目标天数 × 30%，且 < 目标天数 × 60%3 - 4 天最高 75 分
    ≥ 目标天数 × 60%，且 < 目标天数5 - 7 天最高 85 分
    ≥ 目标天数≥ 8 天不额外封顶
    最终付费渠道使用分 = MIN(正常公式得分, 对应封顶值)

    15.4 额度提前用完

    情况处理方式
    额度提前用完，且有真实产出活跃率按实际有效使用天数计算；若使用天数已达目标，活跃率即 100%
    额度提前用完，但使用天数不足活跃率按实际天数计算，不因额度用完而拉满
    额度提前用完，且使用天数集中先按正常公式计算，再受集中消耗封顶约束
    额度提前用完，且无有效产出一致性门控与产出分硬性下限拦截
    异常消耗或疑似刷量进入资源异常分析流程
    16

    Comate 活跃指数

    Comate 活跃指数 = Comate 有效活跃率 × 60% + Comate 强度得分 × 40%

    16.1 Comate 有效活跃率

    Comate 有效活跃率 = Comate 有效使用天数 / 统计周期天数

    有效使用日定义：

    当天有真实产品工作相关使用记录，并达到最低有效使用门槛，才计为有效使用日。

    建议最低有效门槛：

    当天 Comate Token 消耗 >= 组织每人每日 Token 基线 × 20%

    过渡口径：

    如果暂时无法拿到每日明细，可先用 Comate 后台可导出的使用天数作为过渡口径。过渡口径下，每月 Comate 有效使用天数上限为 20 天。

    16.2 Comate 强度得分

    个人周期日均 Token = Comate 有效 Token 总消耗 / 统计周期天数 Comate 强度得分 = MIN(个人周期日均 Token / 组织每人每日 Token 基线, 100%)

    单日峰值截断：

    单日有效 Token = MIN(当天实际 Token 消耗, 组织基线 × 5) Comate 有效 Token 总消耗 = Σ 各日单日有效 Token

    说明：

    1. 强度得分分母采用统计周期天数，避免少数几天集中使用被放大。
    2. 单日 Token 超组织基线 5 倍部分不计入有效 Token。
    3. 强度得分最高 100%。

    16.3 基线校准

    组织每人每日 Token 基线 = 组织全员每人每日 Token 消耗的 P50

    执行要求：

    1. 基线每季度校准一次。
    2. 校准结果需在考核周期开始前公示。
    3. 周期内不调整基线。
    17

    AI 有效产出分

    AI 有效产出分 = 产品工作提效分 + 产品分析质量分 + 可复用资产沉淀分 + 团队复用与影响分
    维度分值考核重点
    产品工作提效20是否减少重复劳动、提升交付效率
    产品分析质量25是否支撑需求、竞品、用户、数据分析
    可复用资产沉淀35是否沉淀 Agent、Skill、Prompt、知识库、SOP 等资产
    团队复用与影响20是否被他人使用，是否进入团队流程

    17.1 产品工作提效，20 分

    得分标准
    0无 AI 辅助工作证据
    5偶尔用于总结、改写、翻译等低复杂度任务
    10能稳定辅助会议纪要、需求整理、PRD 初稿等工作
    15能明显减少重复劳动，有前后对比样例或负责人确认
    20已形成稳定 AI 工作流，能持续提升交付效率

    17.2 产品分析质量，25 分

    得分标准
    0无 AI 辅助分析证据
    8有简单分析材料，但结论较浅或未被使用
    15分析较完整，能支撑需求讨论或评审
    20分析结论被用于需求优先级、方案取舍或版本规划
    25分析显著提升产品决策质量，并沉淀为可复用分析方法

    17.3 可复用资产沉淀，35 分

    子项分值标准
    资产完整度10有明确适用场景、输入、输出、使用说明
    真实业务使用10已用于真实需求、分析、评审、复盘等工作
    可复用性10他人可按说明复用，结果基本稳定
    迭代优化5基于使用反馈持续优化

    可计入资产包括：

    资产类型示例
    Agent用户反馈分析 Agent、竞品研究 Agent、PRD 评审 Agent
    Skill需求拆解 Skill、场景分析 Skill、功能验收 Skill
    Prompt 模板竞品分析 Prompt、PRD 生成 Prompt、用户访谈 Prompt
    工作流从用户反馈到需求池的自动化流程
    知识库产品 FAQ、业务规则库、竞品资料库
    评测集Agent 输出质量评测样例、典型需求样本集
    SOPAI 辅助需求分析流程、AI 辅助版本复盘流程

    17.4 团队复用与影响，20 分

    得分标准
    0仅个人使用，无分享和复用
    5在小范围分享过经验或案例
    10至少 1 名同事实际复用，并有反馈
    15多名同事复用，或进入团队模板库、知识库、流程
    20形成团队级最佳实践，对多人效率或质量有持续帮助
    18

    证据等级与评分上限

    证据等级说明可评上限
    L0 无证据只有口头描述或无法追溯该项最高 20%
    L1 自述证据有个人说明，但缺少样例或链接该项最高 35%
    L2 样例证据有文档、截图、Prompt、Agent、Skill、输出样例等该项最高 60%
    L3 采纳证据进入评审、规划、需求池、方案决策，或被同事引用该项最高 85%
    L4 复用证据被团队成员复用，或进入团队 SOP / 工具 / 模板库该项最高 100%

    执行规则：

    1. 每个维度先按质量评分，再受证据等级上限约束，取两者较小值。
    2. 只有 L3 或 L4 证据，才能拿到该维度高分。
    3. 同一业务事件不得在多个维度重复高分计入。
    4. 每月建议最多提交 3 个代表性案例，不以提交数量取胜。

    项目阶段替代证据

    项目阶段可接受的 L3 替代证据可接受的 L4 替代证据
    探索期机会点、访谈提纲、竞品分析、方案假设被负责人采纳并入库方法或模板被后续项目复用，或被同事用于同类探索任务
    交付期产出进入需求评审、版本规划、需求池或方案决策产出进入团队模板库、SOP，或被多名同事复用
    运营期AI 辅助分析结论被用于优化决策、问题归因或运营动作分析方法、Agent、Skill、看板模板被团队持续复用
    收尾期复盘结论、交接文档、SOP 被负责人采纳并入库交接模板、复盘模板、SOP 被团队后续项目复用
    保密或独立项目脱敏摘要 + 负责人书面采纳记录 + PMO 可核验原始材料脱敏方法论、通用流程或可公开部分进入团队资产库
    替代证据只替代证据形态，不降低质量要求。
    20

    重复认定规则

    1. 重复判定单位为"业务事件"，不是文件。
    2. 每个案例提交时须声明主维度与辅助维度。
    3. 主维度按正常标准评分。
    4. 辅助维度评分上限降为该维度满分的 50%，且须提供独立证据。
    5. 每个案例最多声明 1 个主维度 + 2 个辅助维度。
    6. 若某月无评审或版本节点，可提交跨周期累积证据，但须注明原始产出时间。
    21

    一致性门控与硬性下限

    21.1 一致性门控

    一致性偏差 = |AI 使用行为分 − AI 有效产出分|
    偏差区间处理方式
    < 20 分正常计分
    [20, 25) 分触发预警，校准会议关注
    ≥ 25 分进入一致性校验流程

    一致性校验：

    情况处理方式
    行为分高于产出分 ≥ 25补充产出证据；复核不通过则行为分按 70% 折算
    产出分高于行为分 ≥ 25补充使用过程证据；复核不通过则产出分按 70% 折算

    合理偏差类型：

    类型处理方式
    产出滞后允许在下周期补充采纳或复用证据
    跨周期复用提供历史资产链接和本月复用证据后可正常认定
    项目阶段差异按项目阶段替代证据处理
    渠道统计缺失提供会话、截图、文档版本或负责人确认后人工复核

    21.2 产出分硬性下限

    若 AI 有效产出分 < 40 分，则最终指数不超过 50 分。

    23

    证据脱敏与访问权限

    1. 涉及用户隐私、商业数据、未公开版本规划、竞品敏感信息时，员工应优先提交脱敏摘要。
    2. 原始材料仅向直属负责人、PMO 或指定复核人开放，不在团队公开空间传播。
    3. 证据链接应设置最小可见范围。
    4. 评审记录只保存评分依据和脱敏结论，不复制保存不必要的原始敏感数据。
    5. 如因保密限制无法提交原始链接，可由负责人书面确认"已核验原始材料"，并附脱敏摘要。
    6. 证据包保存周期建议不超过 12 个月；超过周期后仅保留得分、分档和脱敏案例摘要。

    校准、异常与申诉

    25.1 校准机制

    1. 校准抽样不低于团队人数的 20%，且至少覆盖 1 个高分案例和 1 个低分案例。
    2. 初评与校准偏差超过 10 分的案例，进入复评。
    3. 每档分数须勾选对应事实，避免纯印象打分。
    4. 校准会议由 PMO 主持，记录跨团队口径调整决议。

    25.2 异常触发阈值

    信号阈值触发动作
    月初突击消耗月初前 5 个自然日，付费额度使用率 ≥ 80%进入异常分析
    付费集中消耗付费有效使用天数低于目标活跃天数，且付费渠道使用分高于对应封顶值按集中消耗封顶规则修正
    单日 Token 峰值单日 Comate Token 消耗 ≥ 组织基线 × 10 倍进入异常分析
    一致性偏差行为分与产出分偏差 ≥ 25 分进入一致性校验
    产出分硬性下限产出分 < 40 分最终指数封顶 50 分
    L1 证据占比异常团队内 L1 证据占比超过 40%校准会议复核团队口径
    付费资源闲置预警B 类员工单月付费额度使用率 < 20%资源适配预警，不直接扣分
    付费资源连续闲置B 类员工连续 2 个周期付费额度使用率 < 20%，且无合理低适配说明行为分按 90% 折算
    过渡口径依赖过高有效使用天数主要来自"消耗>0"或"有使用即计"标记为试运行风险

    25.3 异常处理流程

    触发 → 通知员工（说明触发信号与依据） → 员工限期补充产出证据或使用过程证据（3 个工作日） → PMO 复核（2 个工作日） → 结论：正常 / 扣分 / 折算 / 扩容 / 移交处理 → 记录归档

    25.4 申诉路径

    1. 员工对异常分析结论有异议，可在收到结论后 3 个工作日内发起申诉。
    2. 申诉由产品负责人与 PMO 联合复核，5 个工作日内给出终局结论。
    3. 申诉期间，原结论暂缓执行，以复核结论为准。
    4. 申诉记录归档，用于校准异常阈值的合理性。

    试运行要求

    试运行启动前必须完成：

    1. 数据采集方案确认。
    2. 过渡口径公示。
    3. 评分量表下发。
    4. 异常阈值确认。
    5. Token 基线首次校准。
    6. 项目阶段标签确认。
    7. 证据脱敏规范确认。

    试运行期间重点观察：

    1. 员工提交证据是否容易核验。
    2. 各团队评分口径是否一致。
    3. 是否出现刷 Token、刷材料、刷资产数量。
    4. Agent、Skill、Prompt、SOP 等资产是否真的被团队复用。
    5. 一致性门控与异常分析是否有效。
    6. 月度评审成本是否可控。
    试运行期建议不直接硬挂绩效。正式纳入绩效前，应完成数据口径、评分量表、申诉机制和证据脱敏机制的验证。
